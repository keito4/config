### 誤検知として棄却した項目（削除しないこと）

- **`pnpm-workspace.yaml`**: 死蔵設定ではない。`minimumReleaseAge: 2880` によるサプライチェーン攻撃対策の意図的な設定。ただし npm 運用との関係をコメントか docs で明文化する（フェーズ 7）。
- **`package.json` の `overrides`**: 推移的依存の脆弱性緩和という正当な用途。削除ではなく「解消条件のメモを付けて定期レビュー」が正しい扱い。
- **`.claude/agents/` の 11 エージェント**: Claude Code がコンテキストに応じて自動選択する正規の仕組みであり「呼び出し機構がない＝死蔵」ではない。ただし規模（4,018 行）の妥当性レビューは行う（フェーズ 5）。
- **ADR 0002**: 初期調査では「Superseded 表記漏れ」とされたが、実際には Status に Superseded by ADR 0006 が明記済み。対応不要。

## 設計原則

1. **各フェーズは独立してマージ可能**。前フェーズの完了を前提にしない項目から着手する。
2. **PR は 400 行 / 25 ファイル以内**（既存ガードに準拠）。1 フェーズ = 複数 PR。
3. **削除が最優先、抽象化は最後**。共通化は「3 箇所目の重複が確認されたもの」だけ行う（YAGNI）。
4. **構造に関わる決定は ADR 先行**（フェーズ 2, 3, 7 に ADR 作成タスクを含む）。
5. **テンプレートリポジトリとしての二面性を明示する**: 「このリポジトリ自身が使うもの」と「下流リポジトリへの配布物（参照専用）」の境界を全フェーズで意識する。
6. **削除・移動は連動物と同一 PR で**: 対象を参照するテスト（Jest / BATS）、README、CLAUDE.md / AGENTS.md の表を必ず同じ PR 内で更新する。本リポジトリのテストはファイル存在を assert するものが多く、削除単体では CI が割れる。

## 全体リスクと中止基準

- 各 PR はマージ前に CI 全緑 + 既存の品質ゲート（pre_git_quality_gates）通過を必須とする。削除・移動 PR は連動するテスト・ドキュメント更新を必ず同梱する（設計原則 6）。
- 削除系タスクで「実は参照されていた」が判明した場合は revert し、参照インベントリ（0-2）を修正してから再実施。
- フェーズ途中で下流リポジトリの破損報告があった場合、該当フェーズを停止し影響範囲を ADR に追記してから再開する。
