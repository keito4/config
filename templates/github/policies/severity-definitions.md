# セキュリティ重大度定義

## 概要

本ドキュメントは、セキュリティ脆弱性の重大度分類と対応SLAを定義します。

---

## 重大度レベル

### Critical（緊急）

**SLA:** 24時間以内に修正

**条件:**

- リモートコード実行（RCE）
- 認証バイパス
- SQLインジェクション（データ漏洩リスクあり）
- 本番環境の秘密鍵・認証情報の漏洩
- 権限昇格によるシステム全体への影響
- 暗号化されていない機密データの露出

**対応フロー:**

1. 即時通知（Slack + メール）
2. インシデント対応チーム招集
3. 影響範囲の特定
4. 緊急パッチまたは緩和策の適用
5. ポストモーテム実施

---

### High（高）

**SLA:** 7日以内に修正

**条件:**

- Stored XSS
- SSRF（Server-Side Request Forgery）
- 権限昇格（限定的な影響）
- 重要データの暗号化不備
- 安全でない直接オブジェクト参照（IDOR）
- セッション管理の欠陥

**対応フロー:**

1. 24時間以内に通知
2. 次のスプリントで対応計画
3. 7日以内に修正をデプロイ
4. 修正後のセキュリティレビュー

---

### Medium（中）

**SLA:** 30日以内に修正

**条件:**

- Reflected XSS
- CSRF（Cross-Site Request Forgery）
- 情報漏洩（非機密データ）
- 依存関係の既知脆弱性（悪用困難）
- 不適切なエラーハンドリング（情報露出）
- セキュリティヘッダーの欠如

**対応フロー:**

1. バックログに追加
2. 30日以内に対応
3. 定期リリースで修正

---

### Low（低）

**SLA:** 90日以内に修正

**条件:**

- ベストプラクティス違反
- 情報漏洩（公開情報のみ）
- 古い依存関係（脆弱性なし）
- コード品質の問題（セキュリティ関連）
- 非推奨APIの使用

**対応フロー:**

1. 技術的負債としてトラッキング
2. 90日以内に対応
3. 定期メンテナンスで修正

---

## エスカレーションポリシー

| 重大度   | SLA超過時のアクション           |
| -------- | ------------------------------- |
| Critical | 即時エスカレーション → CTO/CISO |
| High     | 3日超過で管理職通知             |
| Medium   | 2週間超過でリマインダー         |
| Low      | 60日超過でリマインダー          |

---

## 例外申請

SLA内での対応が困難な場合：

1. 理由の文書化
2. 緩和策の実施（可能な場合）
3. 新しい期限の設定
4. security-team の承認取得

**例外申請テンプレート:**

```markdown
## 例外申請

- **脆弱性ID:**
- **重大度:**
- **発見日:**
- **元のSLA:**
- **希望期限:**
- **理由:**
- **緩和策:**
- **リスク評価:**
```
